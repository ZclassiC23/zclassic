# Next blocker: the 3164483 coin-hole (`refused_spent`) — 2026-06-29

**Status:** OPEN. This is what currently stops the live node from reaching tip,
*after* the commitment-audit wedge was cured (see
[`commitment-audit-wedge-cure-2026-06-29.md`](commitment-audit-wedge-cure-2026-06-29.md)).
**Class:** torn-coins / borrowed-snapshot / sovereign-refold (the long-open hard
problem). **Do NOT band-aid it** — it is consensus-critical.

## How we got here

The commitment-audit cure (this session) decoupled a UTXO-projection heuristic
from `tip_finalize`, so the live node unwedged from H\*=3164075 and **climbed 406
blocks to 3164482**. That exposed a deeper, pre-existing hole the commitment wedge
had masked: the node had never reached 3164483 before.

## The exact mechanism

- `script_validate` at height **3164483** logs `prevout_unresolved tx=dabd990fb94c7c94af5e1e90273dcbf358f0559869d71612696893813bfb8f13 vin=4` — block 3164483 spends a coin **missing from `coins_kv`**. So `script_validate` / `utxo_apply` / `tip_finalize` all pin at 3164483 and H\* holds at 3164482.
- The missing coin was **created at block 3164371** (post-seed; seed/delta horizon = 3155843).
- `coin_backfill` (the auto-repair owner; live `owner_ack=true` via env `ZCL_REDUCER_COIN_BACKFILL_ACK=1`) refuses it. With the *shipped* binary it refuses at the **`creator_in_delta_window`** gate (`app/jobs/src/stage_repair_coin_backfill.c:198`). That gate is a deliberate policy (`docs/work/coin-backfill-repair.md`: "Coins missing inside the delta window indicate a live apply bug → refused").

## What was tried and REVERTED

Relaxing the `creator_in_delta_window` gate to re-derive the coin from its
validated creator block (3164371) — consensus-safe in principle (canonical
hash-verified bytes). The unit-test logic worked, BUT the copy-prove on
`~/.zclassic-c23-wcommit` (a copy that reproduces this exact hole) revealed the
**real deeper blocker**: coin_backfill's G9 no-spend scan then returns
**`refused_spent`** — `[condition:reducer_frontier_reconcile_light] refused coin
backfill h=3164483 status=refused_spent unresolved=1 inserted=0`. The scan finds
the missing coin **also appears SPENT** somewhere in [3164371, 3164482]. Removing
the delta-window gate just exposes this deeper gate. Reverted (it overrode the
documented policy and broke `cb_case_delta_window` without solving the problem).

## The real question to answer first

A coin cannot be spent twice on a valid chain, yet this coin is **both missing
from `coins_kv` AND appears spent** before the block (3164483) that needs it. So
one of these is true:

1. **The coin is genuinely unspent** and the no-spend scan false-positives (e.g.
   it counts a spend in an off-active-chain / orphan block). → Fix the no-spend
   scan (`app/jobs/src/stage_repair_coin_backfill_scan.c`) to only count spends on
   the active chain, then the existing re-derivation path repairs the hole.
2. **The coin is genuinely spent** on this node's chain view, but block 3164483
   (canonical, most-work) references it. → The node's chain view at 3164483 is
   inconsistent (a reorg/fork-view or torn borrowed-snapshot artifact). → Deeper
   coins/chain-view repair; likely the sovereign-refold cutover is the real cure.

**START HERE:** compare this node's view of `dabd990f...:vin4`'s prevout against
`zclassicd` (the consensus reference, kept running):
- find the prevout outpoint: `getrawtransaction dabd990f... 1` → `vin[4].{txid,vout}`;
- ask zclassicd whether that outpoint is unspent at 3164482 (`gettxout txid vout`),
  and which block (if any) spent it.
That single check decides scan-bug (case 1) vs chain-view-corruption (case 2).
