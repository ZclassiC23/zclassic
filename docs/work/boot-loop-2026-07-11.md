# Boot-loop P0 root-cause ‚Äî 2026-07-11 (lane `dev/boot-loop-rootcause`)

## TL;DR

The live node did **not** boot-loop. It booted **once** to `ZClassic C23 node
initialized`, then **stalled at H\*=3176326** because block bodies are missing
from the datadir (`~/.zclassic-c23/blocks/` holds only `blk00000.dat`, while the
block index references files up to `blk00049.dat`). The stall is **loud** (STALL,
`prevout_unresolved`, `stage_repair` flood, sticky_escalator rungs, the
`sapling_anchor_frontier_unavailable` condition all fire).

What *looked* like a "silent boot loop every 6‚Äì30 s" was a background worker
re-running the **full boot open ceremony** on `node.db` every 30 s:

- `payment_processor_thread` (`config/src/boot_background_workers.c:660`, 30 s
  cadence) ‚Üí `store_process_payments` (`app/controllers/src/store_controller.c`)
  ‚Üí **`node_db_open`** ‚Äî which runs `PRAGMA quick_check` (~280 ms, logged
  `[boot] sqlite.quick_check`), schema migration (`db: current schema version
  25`), and the `snapshot_staging_boot_cleanup` / `snapshot_staging_state_boot_cleanup`
  DELETEs. Between cycles the real activity (millions of `read_block_pread` body
  misses) is rate-suppressed ‚Üí the filtered log shows only the boot-shaped
  quick_check/schema/staging sequence repeating ‚áí read as a boot loop.

This is a **direct violation of the "silent halts are unreachable / a stall is
always a named blocker, never a restart" invariant**: a periodic reopen emitted
the exact boot-open log trail with **no statement of who opened the DB or why**.

## Evidence

- `node.log.1` tail (00:55‚Äì01:00, binary `fdf5c1a4c`): one `ZClassic C23 node
  initialized`; thereafter `STALL: h=3176325`, `getheaders planned`, peers
  connected, and the quick_check‚Üíschema‚Üístaging sequence recurring ~every 30 s.
- gdb on a copy running `eef35f2d1` (`break node_db_open`, post-boot hits):
  ```
  #0 node_db_open
  #1 store_process_payments
  #2 <payment_processor_thread>  (start_thread ‚Üí clone3)
  ```
  The **only** post-boot re-opener is `store_process_payments`.
- The 30 s reopen path is **unchanged** between `3b0de63b0` (last-good) and
  `eef35f2d1` (`git log 3b0de63b0..eef35f2d1 -- store_controller.c
  boot_background_workers.c` = empty) ‚Äî i.e. it is **pre-existing**, present in
  both binaries. It is the *silence defect*, not a binary regression.
- **The rollback did not cure the stall.** The current live node is already
  `3b0de63b0` (`~/.local/bin/zclassic23-live`), and `zcl_status` reports
  `height=3176325, header_height=3177172, sync_gap=847, sync_behind=true`, stuck
  at 3176326 with the same sticky_escalator rungs cycling. RPC on 3b0de63b0 is
  responsive; the "RPC server busy" seen on `eef35f2d1` at 00:53‚Äì01:17 could not
  be reproduced on the copy (see gap below) ‚Äî likely lock/threadpool pressure
  from the churn, not a distinct code path.

## Root cause (exact)

`app/models/src/database.c` `node_db_open()` unconditionally runs the boot-only
ceremony (quick_check + `node_db_migrate` banner + snapshot-staging DELETEs).
Every in-process background reopener (`store_process_payments`, and by class the
other `node_db_open` callers) re-runs that ceremony, anonymously, on every
cycle ‚Äî indistinguishable from a boot in a filtered log, and (for the payment
scan) taking a ~280 ms read lock + a staging write-txn on the shared file every
30 s for no benefit.

## Fix (two commits)

1. **Root cause / mechanism** ‚Äî `node_db_open_runtime(ndb, path, reason)` in
   `database.c`: an in-process reopen that **skips** quick_check, the
   snapshot-staging crash-recovery DELETEs, and the schema-version banner (all
   boot-only), and **logs one named line** `db: runtime reopen (reason=‚Ä¶)`.
   `node_db_open()` keeps the full boot ceremony (all existing callers/tests
   unchanged). Convert the three `store_controller.c` opens (the proven 30 s
   offender + the request handlers) to `node_db_open_runtime` with reasons
   (`store.payment_scan`, `store.request`, `store.mark_order_paid`).

2. **Silence generalization** ‚Äî `node_db_open_runtime`'s `reason` is
   **mandatory**: a NULL/empty reason is refused (`LOG_ERR`, returns false), so
   an anonymous background reopen is structurally impossible. Regression test in
   `lib/test/src/test_sqlite.c` asserts (a) a runtime reopen **preserves** the
   `snapshot_staging_*` namespace that a boot open wipes (proves the boot-only
   ceremony is skipped) and (b) the reason argument is enforced.

Follow-up (not in this lane, low risk, mechanical): route the remaining
in-process background reopeners (`blog_controller`, `zslp_service`,
`wallet_view_sync`, `address_backfill`) through `node_db_open_runtime` so no
runtime reopen anywhere re-emits the boot trail. Each must be confirmed
post-boot before converting (a boot-time one-shot must keep `node_db_open`).

The underlying **stall at 3176326 (missing block bodies)** is a separate
datadir-integrity P0, owner-gated to cure (needs a SHA3-checkpoint-bound anchor
snapshot / full-history refold, or re-fetching the bodies). It affects both
binaries and is out of scope for this loop/silence fix.

## Copy-prove flag-parity gap (the reason the earlier copy-prove missed it)

`tools/repro_on_copy.sh` boots the copy with a **different flag set** than the
live systemd service, so a loader-path/flag-specific defect can pass copy-prove
and still wedge live. Concretely the harness:

- **`--light` copy omits `utxo-seed-3155842.snapshot`** (and `blocks/`), and the
  launch **never adds `-load-snapshot-at-own-height`**, `-txindex`, or `-tor`.
  The live service runs
  `-load-snapshot-at-own-height=<datadir>/utxo-seed-3155842.snapshot -txindex
  -tor -nolegacyimport ‚Ä¶`. The earlier copy-prove that "booted fine" never
  exercised the snapshot-loader path at all.

**Proposed contract change** (make copy-prove a valid deploy gate):

1. Add a `--like-live` mode to `tools/repro_on_copy.sh` that reads the effective
   `ExecStart` from `systemctl --user cat zclassic23` (the active drop-in),
   strips only the network-identity/port flags (`-port`, `-rpcport`,
   `-externalip`, `-addnode`, `-connect`), rewrites `-datadir` and the
   `-load-snapshot-at-own-height=` path to the COPY, and passes **everything
   else verbatim** (`-txindex -tor -nolegacyimport
   -load-snapshot-at-own-height=<copy>/‚Ä¶ -showmetrics=0`). Isolated ports and
   HOME are still forced.
2. In `--like-live` mode, force `--full` (or at minimum also copy the snapshot
   file + `blocks/`) so the snapshot-loader path is actually reachable.
3. Also replicate the service `EnvironmentFile` / `Environment=` vars
   (`ZCL_SELF_HEAL_SCAN_DEPTH`, `ZCL_REDUCER_*_ACK`, etc.) into the isolated
   run, and symlink the real `~/.zcash-params` into the isolated HOME (missing
   params otherwise park the node before it reaches the code under test).
4. Deploy gate: a valid copy-prove for a live deploy MUST run in `--like-live`
   mode. The current default (`-connect` dead sink, `-nolegacyimport`,
   `-nofilesync`, no snapshot loader) stays as the fast generic mode but is
   **not** sufficient to clear a live deploy.

### Status: `--like-live` IMPLEMENTED (lane `dev/like-live-copy-prove`)

`tools/repro_on_copy.sh --like-live` now does all four points:

- Reads the effective merged ExecStart via `systemctl --user show zclassic23 -p
  ExecStart --value` (READ-ONLY), drops `argv[0]` (the binary ‚Äî the harness
  always launches the freshly-built candidate `build/bin/zclassic23`), strips
  `-datadir/-port/-rpcport/-fsport/-httpsport/-externalip/-addnode/-connect` and
  the unexpanded `$ZCL_EXTERNALIP_FLAG`/`$ZCL_ADDNODE_FLAGS` refs, and passes the
  rest verbatim (`-listen -txindex -tor -nobgvalidation -nolegacyimport
  -showmetrics=0`).
- Forces `--full` and rewrites `-load-snapshot-at-own-height=` onto the COPY.
- Replicates `Environment=` (via `systemctl show -p Environment`) +
  `EnvironmentFile` (`~/.config/zclassic23/env`) vars, dropping net-identity
  (`ZCL_EXTERNALIP_FLAG`/`ZCL_ADDNODE_FLAGS`), rewriting any live-datadir path
  (e.g. `ZCL_CORE_DIR`) onto the COPY, and overriding
  `ZCL_AGENT_EXPECT_BUILD_COMMIT` to the candidate's short SHA.
- Symlinks `~/.zcash-params` **and** `~/.zclassic` (the tier-1b Sapling-frontier
  borrow reads `$HOME/.zclassic/chainstate`) READ-ONLY into the isolated HOME.

Safety unchanged: the COPY destination is still the fixed
`<HOME>/.zclassic-c23-COPY-<ts>-<slug>` path (no caller-controlled dest), the
live datadir is never written, and a passthrough flag carrying a port/peer/
datadir is refused. Impact rule already maps `tools/repro_on_copy.sh` ‚Üí
`agent_copy_prove` (`app/controllers/include/controllers/agent_impact_rules.def`).
