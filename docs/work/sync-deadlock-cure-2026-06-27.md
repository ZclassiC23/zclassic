# Sync-deadlock root cure — never-stuck class (2026-06-27)

Live wedge: H*=3,162,165 frozen 255 below header tip; tip_finalize `lookahead_tip_missing`.
Design from workflow wqzxc830g (3/5 mappers + synthesis; robust to both body-absent and frozen-ok=0 sub-cases).

## Unified root cause
A single circular dependency: forward block FETCHING is gated on the published provable tip H*, while advancing H* at the frontier is gated on the body/verdict of the lookahead successor that only forward fetching can deliver. Concretely, tip_finalize publishes H*=N only when successor N+1 is itself body-present and foldable (a body-level lookahead, not a header-level one), and active_chain_height()==H* is the anchor for msg_get_height, gap_fill's tip base, and the stall-recovery probe. So a single block above H* that is body-missing or carries a frozen ok=0 verdict simultaneously (1) deadlocks tip_finalize's lookahead, (2) starves the H*-anchored fetch window of the parallelism needed to bulk-pull the heights that would resolve the lookahead, and (3) trips a self-heal whose witness (request-counter advance) is unsatisfiable when no body is fetchable from the anchored window, latching to operator_needed. The defect is that the lookahead consumes only header-level fields of N+1 (pprev, nChainWork, phashBlock, nHeight) yet GATES on N+1's body/script validity, and the download window is pinned to the fold tip rather than the header tip.

## Invariants (must preserve)
- H* never exceeds the MIN over every success-checked log's contiguous ok=1 prefix (reducer_frontier.c:558-596); never stamp ok=1 to raise H*. A verdict row is written only by the owning stage after it actually ran.
- tip_finalize may only finalize N when N's OWN upstream verdict is ok=1 (utxo_apply_log_at(N).ok==1, tip_finalize_stage.c:443-458) AND the cursor invariant next_h < uv_cursor holds (utxo_apply folded past N). Never publish a height whose own body is unvalidated; never publish N+1.
- tip_finalize must reflect the real most-work chain: finalize N->N+1 only when new_tip->pprev==old_tip, new_tip is BLOCK_VALID_HEADER (PoW verified), new_tip->nChainWork strictly greater, and new_tip is an ancestor of pindex_best_header. A non-canonical / lighter / header-invalid successor still holds (chainwork_not_greater stays an ok=0 advance, reorg_detected branch intact).
- The reorg-rewind machinery (rewind_cursor_if_active_chain_reorged + tf_refresh_provable_tip, tip_finalize_stage.c:335-394) must stay so any finalize a deeper reorg invalidates is retractable and H* is lowered at the chokepoint.
- The phantom-anchor guard (reducer_frontier.c:549-550, coins_kv_is_proven_authority) and the HARD GUARD hs<anchor->hs=anchor must remain untouched.
- A re-derivation cure must ACTUALLY re-run consensus validation against the body (hash-bound) and REFUSE (typed blocker, no stamp) if the dry-run finds the height invalid; never blindly stamp ok=1 or extend the window past an unproven height.
- Zero changes to Equihash params, activation heights, or block/tx validity predicates: E13 check-consensus-parity and test_consensus_parity stay green; the fix is liveness/scheduling only.
- Served tip safety: getblockcount / P2P start_height keep serving H*; de-anchoring affects only what we REQUEST, never what we SERVE or advertise as validated.

## Fix steps
### Step 1: De-anchor the steady-state and stall fetch window from H* to the header tip
- **file:** app/services/src/block_sync_service.c
- **change:** In syncsvc_build_stall_recovery (around lines 238-248) and the steady-state queue path, originate the fetch/probe window at min(active_chain_height, body_fetch_stage_cursor, body_persist_stage_cursor) rather than active_chain_height alone, and widen the probe ceiling toward pindex_best_header instead of our_h+STALL_PROBE_WINDOW so every header-known/body-missing height in (fold_frontier .. header_tip] is reachable. Reuse gap_fill's existing pattern (gap_fill_service.c:176-180 already lowers tip_h to body_fetch_cursor-1) so the two fetch planners agree on the same base. Keep the existing dedup/in-flight filters.
- **why:** The stall recovery and steady queue are pinned to H* (active_chain_height), so they loop 'getdata: 1 block' just above H* and never bulk-pull the heights whose arrival is required to advance the lookahead. Lowering the base to the body-stage cursor (the real fold frontier) and raising the ceiling to the header tip lets the node fetch the whole [frontier..header_tip] span in parallel.
- **risk:** LOW — changes only what is requested, not what is served. Bounded by existing dl_queue capacity + dedup. Must keep the in-flight/HAVE_DATA filter so it does not re-request present bodies.

### Step 2: Make gap_fill the single forward-fetch authority and confirm it spans frontier..header_tip every 5s
- **file:** app/services/src/gap_fill_service.c
- **change:** No new window math needed (lines 162-180 already base tip_h on min(active_chain_height, body_fetch_cursor-1) and scan to best_header over block_map ancestry, skipping HAVE_DATA). Add: also lower the base to min(body_persist_cursor-1, tip_finalize_cursor) so a frozen tip_finalize cannot leave a hole the steady queue ignores; and emit a one-line EV when the computed window is non-empty but enqueued==0 for >N ticks (real 'fetchable gap exists but nothing queued' signal that download_queue_starved can consume as its trigger).
- **why:** gap_fill already de-anchors from H* and walks header ancestry, so it is the correct mechanism; it just needs its base lowered to cover a frozen finalize tip and an honest 'gap exists, nothing queued' signal so the self-heal in step 5 can witness real starvation rather than misfire.
- **risk:** LOW — additive lowering of the scan base over the same block_map walk; the HAVE_DATA skip prevents re-requesting present bodies.

### Step 3: Finalize N on a HEADER-only canonical-successor witness (break the body-level lookahead deadlock)
- **file:** app/jobs/src/tip_finalize_stage.c
- **change:** At the transient_reason gate (lines 528-554), when precondition_block_reason(new_tip) is non-NULL for the lookahead reasons (have_data_missing / not_script_valid / not_header_valid / block_missing) BUT (a) N's own utxo_apply verdict already read at line 443 is ok=1, AND (b) new_tip is the CANONICAL header successor — new_tip->pprev==old_tip (already guaranteed here) AND new_tip carries BLOCK_VALID_HEADER (PoW verified) AND new_tip->nChainWork > old_tip->nChainWork AND block_index_get_ancestor(ms->pindex_best_header, next_h+1)==new_tip — then FINALIZE N: insert the finalized ok=1 row keyed at height N with N's OWN phashBlock/nHeight (NOT N+1), refresh H* via tf_refresh_provable_tip, and advance the cursor to N+1. Retain the existing JOB_IDLE hold ONLY when new_tip is NOT yet a header-valid canonical successor (genuinely missing header / not on best_header / not greater work). Leave the chainwork_not_greater ok=0-advance (lines 555-565) and reorg_detected branch (496-505) unchanged.
- **why:** Finalizing the N->N+1 linear transition consumes only N+1's HEADER fields (pprev line 496, nChainWork lines 555/567, phashBlock/nHeight lines 591/628); post-finalize side effects already treat a missing N+1 body as a benign skip (tip_finalize_post_step.c:63-81). N is independently proven (its own ok=1 across all upstream stages). So requiring N+1's BODY to finalize N is an unnecessary gate that creates the deadlock. Using N+1 only as a header-level most-work successor witness lets H* climb to N, which (with steps 1-2) un-anchors the fetch window so the pipeline gets the bodies it needs. N+1 itself is NOT finalized — its cursor still requires its own upstream ok=1 before tip_finalize advances past it.
- **risk:** MEDIUM — the load-bearing change. Safe because it never publishes N+1, never serves an unvalidated height, and the most-work/PoW/ancestor predicate is the same evidence the active-chain window is built from. The reorg-rewind path makes it retractable. Requires re-pinning the test contract (step 6).

### Step 4: When N+1's body is present but its upstream verdict is a frozen terminal ok=0, route to re-derivation instead of abstaining
- **file:** app/conditions/src/reducer_frontier_reconcile_light.c
- **change:** At the pending_forward early-returns (detect lines 820-824, remedy lines 889-900), abstain ONLY when the successor verdict is genuinely still in flight (the owning stage's cursor <= refused_height — the pipeline has not yet decided that height). PROCEED into the existing hash-bound re-derivation path (stage_reducer_frontier_reconcile_light, which re-runs validation from the body and refuses with stale_script_repair_genuinely_invalid if the dry-run is still invalid) when the cursor has PASSED refused_height with a terminal ok=0. Tighten stage_repair_tipfin_refusal_is_pending_forward (stage_repair_reducer_frontier_tipfin.c:175-187) accordingly so 'pending' means cursor-not-yet-arrived, not 'cursor arrived with ok=0'.
- **why:** Covers the task-evidence sub-case (utxo_apply wrote ok=0 at 3162167, coins_applied_height=3162167). Today reconcile_light treats any pending_forward as a self-resolving transient and opts out, but a terminal ok=0 never resolves — so the one repair path that could re-derive the verdict abstains. Re-running validation from the body is the sanctioned cure (self-verified-tip-plan Act 1). With step 3, a re-derived ok=1 at N+1 also lets tip_finalize advance normally.
- **risk:** MEDIUM — touches verdict state, but only via the existing hash-bound dry-run that REFUSES on genuine invalidity (no blind stamp). Must keep the peer-lag gate and copy-prove. If the body is genuinely absent this path no-ops and steps 1-3 carry the cure.

### Step 5: Make download_queue_starved fire only on a real fetchable gap and re-arm instead of latching to operator
- **file:** app/conditions/src/download_queue_starved.c
- **change:** (A) In detect_download_queue_starved (lines 41-62), after dl_get_stats, ALSO require evidence of a real fetchable body gap: best_header_height > fold_frontier AND at least one header-known height in (fold_frontier..best_header] lacks BLOCK_HAVE_DATA and is not BLOCK_FAILED_MASK (reuse gap_fill's gap detector / the step-2 'gap exists, nothing queued' signal). If no fetchable body gap exists, return false and clear g_first_seen — this is not download starvation. (B) In the struct (lines 95-108) add .cooldown_secs=300 and .cooldown_max_rearms=0 (unbounded) so the max_attempts=5 cap re-arms every 5 min (condition.c:216-251) instead of permanently latching to EV_OPERATOR_NEEDED, matching peer_floor_violated.c.
- **why:** (A) makes the condition's request-counter witness always satisfiable when it fires (a real fetchable body exists), so it can never reach operator_needed by construction on a bodies-present wedge. (B) belt-and-suspenders: even an unforeseen misfire re-arms rather than dead-ends the owner. Together they remove the false operator page for the entire class.
- **risk:** LOW — touches only self-heal triggering/escalation cadence; no verdict written, no tip served, no consensus predicate. cooldown re-arm is a proven pattern.

### Step 6: Re-pin the test contract that asserts the deadlock-hold
- **file:** lib/test/src/test_tip_finalize_stage.c
- **change:** Replace the assertion at lines 858-886 (currently: tip_finalize holds the predecessor at JOB_IDLE indefinitely when the successor lacks HAVE_DATA) with a finalize-on-canonical-header-successor assertion: given N upstream-ok=1 and N+1 a header-valid most-work canonical successor (BLOCK_VALID_HEADER, greater work, best_header ancestor) but body-missing, step_finalize ADVANCES the cursor to N+1, publishes (N, hash(N)), and raises H* to N; AND a NEGATIVE case: when N+1 is NOT a canonical header successor (missing header / not on best_header / not greater work) it still HOLDS at JOB_IDLE. Add a coverage test that download_queue_starved does NOT fire when all (fold_frontier..header_tip] heights have HAVE_DATA, and DOES (then re-arms, never latches) when a real body gap exists.
- **why:** Step 3 is an intentional behavior-contract change; the existing test pins the old deadlock and would false-red. The new tests pin the never-stuck contract and the negative guards so a future regression that re-introduces the body-level lookahead or the latch is caught.
- **risk:** LOW — test-only, but it is a deliberate contract change and must be called out (not a silent test edit).

## Generalizes
The class is 'stage S holds a fully-upstream-proven height N because its forward witness (the lookahead successor N+1) is body-missing or carries a terminal ok=0 verdict, and the H*-anchored download window then starves the fetches that would resolve the witness.' The combined fix dissolves the class structurally rather than unsticking 3162166: (1) The fetch window is keyed off the fold FRONTIER and ranges to the HEADER tip (steps 1-2), so any height in (frontier..header_tip] that lacks a body becomes fetchable in parallel — the node can never sit below a known header tip while bodies are fetchable, for any height or any number of stuck blocks, with no height/hash/reason string referenced. (2) tip_finalize advances on N's own ok=1 plus N+1's HEADER-only canonical-successor evidence (step 3), so H* always tracks the deepest UPSTREAM-proven height and is never one block behind the other four stages waiting on a body — both lookahead_tip_missing and successor_pending sub-cases use the same code path and both stop freezing H*. (3) The body-present-but-frozen-ok=0 sub-case routes to the existing hash-bound re-derivation (step 4) instead of being misclassified as a self-resolving transient. (4) The self-heal can only fire when a real fetchable gap exists and re-arms instead of latching (step 5), so the operator_needed escalation for this entire class disappears — progress resumes automatically and the witness is always satisfiable. The genuine guards are untouched: a non-canonical / lighter / header-invalid successor still holds; a genuinely invalid body still refuses with a typed blocker; H* never climbs past the contiguous-ok prefix.
