# Miner-signaled Equihash 200,9 sidegrade (no fixed hardfork height)

## Context

The chain has run Equihash 192,7 (400-byte solutions) since the Bubbles upgrade at height 585,318. The goal is to return to 200,9 (1344-byte solutions) **without a flag-day activation height**: the validator itself tallies miner signaling in block versions and locks in once ≥51% of blocks signal sustained over ~1 month; activation then follows after a **~1-month grace period (34,560 blocks)** that serves as the deterministic coordination runway for pools, exchanges, and miners — the exact activation height is known and machine-readable a month in advance. This is feasible and fits the codebase: `nVersion` is stored in every `block_index` entry (cheap header-only ancestor tallies via `pprev`/`pskip`), all `nVersion` checks here and in legacy zclassicd/MagicBean are `>= 4` (a signal bit relays through the existing network), PoW *verification* is already solution-size-dispatched (1344 → 200,9 verifies today), and the single consensus gate pinning which size is allowed at a height is the expected-size check in `contextual_check_block_header`. No versionbits machinery exists anywhere in the tree — it gets built as a new module. This feature ships in the first release, so every C23 node carries it from day one.

**Terminology — this is a sidegrade, not an upgrade.** Returning to 200,9 is a lateral parameter move: the new parameters are not technically superior to 192,7, and the switch trades one security model for another (rentable-GPU risk for redirectable-ASIC risk — see the Implications section). Where identifier names contain "upgrade" (`ehUpgrade`, `eh_upgrade`, `-signal-eh-upgrade`), the word is deployment vocabulary inherited from the network-upgrade tables, not a claim of improvement.

**User decisions:** sustained-majority tally (8 consecutive 4,320-block windows, each ≥51% signaling → LOCKED_IN); BIP8-style no expiry; ~1-month grace (34,560 blocks) between LOCKED_IN and ACTIVE; graduated difficulty relief at the switch (reuse `scaleDifficultyAtUpgradeFork`); built-in miner signals and falls back to the generic solver for 200,9. All work goes on a new branch; this design document is committed into the repo (`docs/design/equihash-200-9-versionbits.md`) as part of the PR.

## State machine (deterministic from headers only)

- Signal: `nVersion & (1<<16)` (miners set `nVersion = 4 | (1<<16)` = 0x10004 — verified safe against every nVersion read in the tree: serialization is lossless int32, Equihash seal commits all 4 bytes, no equality/switch checks anywhere).
- Fixed windows of W blocks aligned to height multiples; window passes if ≥ threshold blocks signal. Streak of consecutive passing windows; a failing window resets it. Streak reaches N → **LOCKED_IN** (sticky). **ACTIVE** at `H_a = locked_in_boundary + grace`. From `H_a`, expected params = 200,9.
- Evaluation is recursive per window boundary with a mutex-guarded in-memory cache keyed by **boundary block hash** (reorg-safe by construction; no invalidation needed). Full recompute after restart ≈ 730 boundaries × 4,320 pprev hops — milliseconds.
- Mainnet: W=4,320 (~3.75 d at 75 s), threshold=2,204 (51.02%), consecutive=8 (≈1 month), grace=34,560 (=8·W, so H_a is a boundary). Testnet: 1,440/735/4/2,880. Regtest: 16/9/4/32 with the "upgrade" flipping 48,5 → 96,5 (68-byte solutions; both sizes already in the verifier's dispatch map) so `generate`-driven e2e works.

## Implications of returning to 200,9: ASICs

Equihash 200,9 is the dominant ASIC-mined Equihash variant (Zcash and its forks; Antminer Z9/Z11/Z15-class hardware). Bubbles' move to 192,7 at 585,318 was an ASIC-resistance decision — 192,7 has no commodity ASICs and is mined by GPUs. Reverting re-opens the chain to the existing 200,9 ASIC fleet:

- **Hashrate and security:** the chain joins a vastly deeper hashrate market. ZCL's thin GPU hashrate has historically been cheap to out-rent (the 2018 51% double-spend attacks); ASIC depth raises the absolute cost of majority attacks, but idle 200,9 ASIC capacity elsewhere (Zcash et al.) is large relative to whatever ZCL attracts, so a hostile redirect of even a small fraction of that fleet remains a realistic attack vector until ZCL's own 200,9 difficulty grows. Net: security model shifts from "rentable GPU" risk to "redirectable ASIC" risk.
- **Miner displacement:** current 192,7 GPU miners become uncompetitive at `H_a` essentially instantly. Note the governance tension this creates for the signaling itself: the 51% tally is cast by the *current* GPU miners/pools, who would be voting to obsolete their own hardware — reaching threshold realistically requires explicit pool-operator coordination, not just default-on node signaling.
- **Difficulty shock both directions:** if ASICs pile in at `H_a`, blocks come fast until the 17-block averaging (clamped ±16%/25% per block) catches up — bounded, self-correcting. If instead GPU miners leave at `H_a` faster than ASICs arrive, blocks slow — that direction is what the graduated fork-scaling relief (step 5) exists for. Both are transient and bounded by existing mechanisms.
- The solution-size change (400 → 1344 bytes) adds ~1 KB per header; bandwidth/storage impact is negligible.

## Exchange and pool adoption (the grace period is the coordination window)

The ~1-month grace between LOCKED_IN and ACTIVE is the deliberate coordination runway, and it is *deterministic and machine-readable*: the moment LOCKED_IN occurs, `getdeploymentinfo` / `zcl_state versionbits` publish the exact `active_height` (`H_a`) about a month in advance. Infrastructure can automate against it.

- **Pools:** must swap stratum/solver backends from 192,7 to 200,9 work at exactly `H_a`. `getblocktemplate` from an upgraded node carries the correct version bits and (post-`H_a`) the new parameters automatically, so template-driven pools need only schedule the solver-side switch during grace. Pools that miss the switch produce invalid (wrong-size) work from `H_a` onward and lose all shares — the month-long, block-height-precise notice is what makes this a scheduled maintenance instead of an emergency.
- **Exchanges:** need an upgraded node before `H_a` and should (a) raise deposit confirmation requirements through the transition window (hashrate handover = elevated reorg risk), and (b) ideally pause deposits/withdrawals for a window around `H_a` itself. Exchanges running legacy `zclassicd` are size-permissive (it derives Equihash params from solution size) and will follow the most-work chain either way — but they get none of the new enforcement, so upgraded nodes are still the recommendation.
- **Legacy zclassicd/MagicBean nodes follow either way:** their permissive size dispatch (Equihash params derived from solution size) and `nVersion >= 4` rule mean they relay and accept both eras' blocks and track the most-work chain across `H_a`. Every C23 node carries this feature from the first release, so there is no older C23 population to fork off.
- **Communication milestones the design gives for free:** (1) deployment ships → "signaling open"; (2) streak visibly building in `getdeploymentinfo` → ecosystem heads-up; (3) LOCKED_IN → hard one-month countdown with exact height; (4) `H_a` → switch. Each is a height, not a date — publish the 75-second-spacing date estimate alongside.

## Risks (stated honestly)

- **Fast-sync tails:** a node snapshot-syncing past a future H_a can't walk the windows until headers backfill — falls back to table params until then; once mainnet locks in, ship a checkpoint row at H_a as belt-and-braces.
- **Legacy network claim** ("MagicBean relays 0x10004 blocks, and zclassicd accepts 1344-byte solutions via its size-dispatch") is consistent with both codebases' sources but is an external-network assumption — verify against a live zclassicd before any mainnet signaling campaign.
- `equihash_basic_solve` at 200,9 is slow — in-process mainnet mining stays impractical (external miners via getblocktemplate are the real path).
- `mine_block_pow` signature change ripples into 3 test files (mechanical, same commit).
