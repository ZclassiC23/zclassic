# ADR-0002: Physically split and seal the consensus core (`core/`)

- **Status:** Accepted and landed 2026-07-10 (`check-core-seal` flipped HARD
  at commit `7ae4ab55e`, "core: freeze the consensus-core seal — flip
  check-core-seal HARD (W5)").
- **Deciders:** Project maintainer.
- **Related:** `docs/FRAMEWORK.md` (the eight-shape architecture this split
  sits inside), [`docs/ZVCS.md`](../ZVCS.md) (the in-binary VCS whose own
  seal guard mirrors this one), `core/UNSEAL.md` (the append-only owner
  unseal ritual log).

---

## Context

`docs/FRAMEWORK.md`'s Prime Directive draws a hard line: Equihash parameters,
activation heights, block/transaction validity, and coin accounting are
**consensus-parity territory** — a change here must never ship to zclassic23
ahead of the reference chain, even framed as opt-in or miner-signaled (see
`docs/CONSENSUS_PARITY_DOCTRINE.md`). Before this ADR, that line was
enforced entirely by *review discipline*: the consensus predicates and
static parameter tables lived scattered across `domain/consensus/`,
`lib/consensus/`, `lib/chain/`, and `lib/core/`, alongside ordinary
application code, with no structural barrier stopping an autonomous dev-loop
cycle from editing a validity predicate and auto-publishing it the same way
it would publish an explorer template fix.

The project increasingly relies on autonomous and semi-autonomous agent
cycles (the dev loop described in `docs/work/HOTSWAP.md`, ZVCS's auto-anchor
in `docs/ZVCS.md`) to move fast on everything *outside* consensus. "Code
fearlessly, not recklessly" requires that fast path to be **structurally**
incapable of touching consensus rules without an explicit, logged, owner-run
step — not merely discouraged from it by a doc or a code-review convention
that an agent (or a bug) can route around.

## Decision

Physically move the consensus predicates and static, height-keyed parameter
tables into one top-level tree, `core/`, and pin its byte-integrity with a
SHA3-256 manifest that the build refuses to ignore.

### 1. The physical split

`core/` holds four contexts, each a pure rename/absorption of existing code
(no logic changes), each keeping its pre-split `#include` token so callers
did not need touching:

| Context | Moved from | Include token preserved |
|---------|-----------|--------------------------|
| `core/consensus/` | `domain/consensus/` | `-Icore/consensus/include` → `domain/consensus/*.h` |
| `core/params/` | `lib/consensus/` | `-Icore/params/include` → `consensus/*.h` |
| `core/math/` | `lib/core/` (pure math) | `-Icore/math/include` → `core/*.h` |
| `core/chainparams/` | the pure params/verify subset of `lib/chain/` | `-Icore/chainparams/include` → `chain/*.h` |

(See the `CORE_CONTEXTS` / `CORE_INCLUDES` block in `Makefile`, just above
the `core-seal` targets, for the exact `-I` wiring.) Every `core/*.c` under
these four `src/` directories is compiled into `ALL_SRCS` exactly as before
— **`core/` stays in the single whole-program LTO link**; this is a
directory reorganization plus a build-time seal, not a new library, a
separate compilation unit, or a process boundary. Landed as five pure-rename
commits: `b4e1b7d91` (consensus), `e44b2d44e` (params), `fe617bce6` (math),
`f5b0b2b0b` (chainparams), plus `4e4f90a29` (redirecting two boundary
leaks downward pre-seal).

### 2. The seal

`core/MANIFEST.sha3`, generated by `tools/core_seal.c` (a standalone
build-time tool: no external deps, links only the in-tree FIPS-202 SHA3-256
and `memory_cleanse`), records a SHA3-256 digest of every tracked file under
`core/` plus a single `ROOT` digest over the sorted `(path, filehash)`
stream. `make core-seal` (re)writes it; `make core-seal-check` / the `make
lint` gate `check-core-seal` verifies it. **`check-core-seal` is HARD**: any
drift in `core/` from the recorded manifest fails the build. It reads the
file list from `git ls-files -z core/` — the same publish-bridge git
described in `docs/ZVCS.md`, invoked only from the Makefile, never from
inside `tools/core_seal.c` itself.

A second gate, `check-core-include-boundary`
(`tools/scripts/check_core_include_boundary.sh`), enforces that `core/` may
not depend upward or sideways — in particular, not on `lib/validation/` —
the same discipline `check_domain_purity.sh` already applies to `domain/`.

### 3. The owner-only unseal ritual

**Sealed ≠ frozen.** Consensus-parity fixes still ship routinely (per
`docs/CONSENSUS_PARITY_DOCTRINE.md`) — the seal does not block a deliberate,
reviewed change; it blocks an *unattended* one. The ritual, recorded
append-only in `core/UNSEAL.md`:

```
make core-unseal REASON="why this consensus-core change is needed"
#   → appends a dated entry to core/UNSEAL.md (old ROOT hash + reason)
#   → writes .core-unseal-token (gitignored), honored for exactly one commit
... make the sealed-core edit ...
make core-seal        # re-freezes the manifest, consumes the token
make lint && make test_parallel   # must land green, incl. test_consensus_parity
git commit             # the reseal + the edit land together
```

`make core-unseal` requires a `REASON` argument and is an owner-run `make`
target — no agent source edit can mint `.core-unseal-token` as a side
effect of a normal commit. (`core/UNSEAL.md` documents v1.1 as upgrading
this to an ed25519 owner signature so consent cannot be forged; that is
future work, not yet landed.)

### 4. The dev-loop refusal (the part that actually blocks the running binary)

The seal alone only gates `make core-seal-check` / `make lint`, which run at
human-reviewed-commit cadence. The autonomous fast loop
(`tools/dev/devloop_cycle.c` / `tools/dev/devloop_plan.c`) runs far more
often than that and needed its own pre-publish check. `zcl_devloop_plan_files()`
classifies every changed file; `zcl_devloop_path_is_sealed_core()` matches
the same surface `core/MANIFEST.sha3` pins (a single `core/` prefix check,
deliberately broader than the older `path_is_consensus_risk()` prefix list
so there is only one place this can drift). `zcl_devloop_run_cycle()`
checks `plan.sealed_core` **before any hot-swap or reload publish step**:

- No valid `.core-unseal-token` present → `emit_refusal()` writes a
  structured `zcl.dev_cycle.v1` envelope (`"status":"refused",
  "reason":"sealed_consensus_core"`, the exact sealed paths that tripped it,
  and `"unseal":"make core-unseal REASON=..."`) to stdout and to the
  persisted verdict file, and the process returns CLI **exit code 3**
  (blocked-by-precondition) — the cycle never reaches the publish step at
  all, so the running binary is never touched.
- A valid token present → the seal is lifted for *this* cycle, which is
  simultaneously marked `consensus_risk` (so it routes to the heaviest
  proof path, the same treatment a `lib/validation/` edit gets today). The
  token is consumed by `make core-seal`, not by the dev-loop check itself,
  so one `make core-unseal` authorizes exactly one landed commit — which may
  span several iterative dev-cycles while the fix converges — never just one
  cycle.

This refusal landed at commit `070849e7f` ("dev: fast loop structurally
refuses to auto-publish sealed consensus core"), Wave 2.4 of the
power-station plan. ZVCS's own independent seal guard
(`lib/vcs/src/vcs_seal.c`, described in `docs/ZVCS.md`) is a second,
defense-in-depth check at the source-snapshot layer; as of this writing it
runs *after* the dev-loop's own publish step (advisory only at that
integration point) — the dev-loop refusal above is the one that is
load-bearing for blocking the running binary.

## Consequences

**Positive:**

- An agent cannot silently change consensus validity rules through the
  autonomous fast path — the refusal is structural (a build-time seal +
  a pre-publish file-classification check), not a convention a prompt or a
  reviewer has to remember to enforce.
- `core/` becomes a single, greppable answer to "is this file consensus?" —
  the same one-`core/`-prefix check drives the seal manifest, the include
  boundary gate, and the dev-loop refusal, so the three cannot drift apart
  from each other by construction.
- The split cost zero runtime behavior change (pure renames + include-token
  preservation); `core/` stays inside the single whole-program LTO link, so
  there is no new ABI boundary, no new binary, and no performance
  regression from the reorganization itself.
- The owner-only unseal ritual keeps a clear, append-only audit trail
  (`core/UNSEAL.md`) of every deliberate consensus-core change, distinct
  from ordinary commit history.

**Negative / Risk:**

- `core/` is a coarser-grained seal than the prior `path_is_consensus_risk()`
  prefix list: it now also covers `core/math/` (pure arithmetic helpers that
  are not directly validity predicates), which means some non-consensus
  edits inside `core/` will also trip the refusal and require the unseal
  ritual. This is a deliberate false-positive-tolerant choice ("no second
  list to drift" beats a narrower but driftable boundary).
- The unseal token is a plain gitignored file in v1, not a cryptographic
  signature — an operator with local write access to the repo (not an
  autonomous agent following the fast-loop-only publish path) could create
  it by hand. The ADR accepts this because the fast loop itself never mints
  the token; v1.1's ed25519-signature upgrade is the follow-up hardening
  step, not yet built.
- `check-core-include-boundary` is a text-pattern boundary gate (grep-based,
  like `check_domain_purity.sh`), not a compiler-enforced module boundary;
  a sufficiently indirect include could in principle evade it. Accepted for
  the same reason the domain-purity gate is accepted elsewhere in this
  codebase — it is a HARD ratchet with an empty baseline, so any new
  violation fails the build immediately.

## Alternatives considered

**(a) Lint-only boundary, no physical move.** Keep the consensus code where
it lived (`domain/consensus/`, `lib/consensus/`, `lib/chain/`, `lib/core/`)
and rely solely on a `check-consensus-parity`-style gate scanning a hardcoded
path list. Rejected: a hardcoded path list is exactly the kind of thing the
lint-gate-hollowness audit (`docs/work/lint-gate-hollowness-audit.md`) found
goes silently stale — a new consensus file added under an existing directory
but outside the listed sub-paths escapes the gate with the build still
green. A physical `core/` tree makes "is this consensus" a single glob
(`core/`) instead of a maintained list that can drift out of sync with the
actual consensus surface.

**(b) Separate static library, severing the LTO link.** Build `core/` as its
own `.a` and link it in, so the consensus surface has a real ABI/object-file
boundary in addition to a source-tree one. Rejected for this wave: it would
give up the whole-program LTO the rest of the binary relies on for the
`core/` object files specifically (cross-TU inlining, dead-code elimination
across the consensus/application boundary), for a boundary the seal +
include-check already enforce at build time without paying that cost. Worth
revisiting only if a future wave needs `core/` to be independently
versioned or shipped standalone (it does not, today — one binary, one
sealed core).

**(c) Runtime capability check instead of a build-time seal.** Gate
consensus-affecting RPC/MCP calls behind a runtime permission model instead
of sealing the source tree. Rejected: this ADR is about the *source-editing*
surface (an agent's dev-loop cycle), not the *runtime* RPC surface — those
are different threat models (see `docs/SECURITY_AND_INTEGRITY.md` for the
runtime-facing controls). A runtime check does nothing to stop an
autonomous agent from committing and hot-swapping a changed validity
predicate; only a build/publish-time gate does.
